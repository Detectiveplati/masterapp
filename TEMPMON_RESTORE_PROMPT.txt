MODULE PURPOSE
Recreate an equipment temperature monitoring module named TempMon for a central kitchen master app. It monitors freezers, chillers, and warmers using LoRa/TZONE sensor tags, stores readings, raises HACCP alerts, tracks corrective actions and calibrations, supports browser push notifications, provides dashboards and reports, and integrates with the Food Safety monthly checklist/report system.

URL AND PERMISSION MODEL
- Static frontend lives at /tempmon and requires the user to have tempmon permission.
- API router is mounted at /api/tempmon.
- Auth is cookie/JWT based via existing requireAuth middleware.
- Most TempMon routes require authenticated users.
- Admin-only routes require requireAdmin.
- The monthly confirmation endpoint requires requirePermission('tempmon').
- Push notifications send to users/subscriptions with tempmon permission.
- The LoRa device management UI uses /templog/api/lora/* but is presented inside the tempmon setup page and should require tempmon access.

LORA/TZONE GATEWAY INTEGRATION UNDER /templog/api/lora
This is essential. The gateway management is implemented in server.js using the TempLog DB and forwards mapped sensor readings into TempMon.

Device registry routes:
- GET /templog/api/lora/devices: list registered LoRa devices from templog_lora_devices.
- POST /templog/api/lora/devices: register sensorId/model/equipment/alias/notes/enabled/tempmonUnitId; validate model TAG07/TAG08B/TAG08L/TAG09; if tempmonUnitId present, sync/create TempMonDevice.
- PUT /templog/api/lora/devices/:sensorId: update mapping, enable/disable, model, alias, notes, tempmonUnitId; sync TempMonDevice active/unit.
- DELETE /templog/api/lora/devices/:sensorId: delete mapping.

Gateway troubleshooting routes:
- GET /templog/api/lora/tcp-log?format=json|text&limit=N: in-memory TCP log.
- GET /templog/api/lora/tcp-config: returns host/port for gateway UI using LORA_TCP_PROXY_* or Railway TCP vars, fallback LORA_TCP_PORT.
- GET /templog/api/lora/events?limit=N: raw gateway receive records from templog_lora_gateway_events.
- GET /templog/api/lora/status?scan=N: latest status for registered sensors by scanning recent gateway events.
- GET /templog/api/lora/discover?scan=N&hours=N: sensors seen but not registered.
- POST /templog/api/lora/receive: HTTP gateway receive endpoint. Validate optional LORA_HTTP_TOKEN. Extract tag rows from payload.data.tag07/tag08b/tag09/etc. Normalize sensor IDs. Insert raw event. Forward mapped/enabled readings into TempMon and also insert legacy equipment temp readings.

LoRa parsing behavior:
- Sensor IDs are normalized to uppercase and numeric IDs shorter than 8 digits are left-padded to 8 digits.
- Supported model aliases normalize to TAG07, TAG08B, TAG08L, TAG09.
- Accept payloads where data contains arrays named tag*.
- Fallback to top-level arrays only if no data envelope exists.
- Accept temperature from temp/temperature/Temperature/Temp/T.
- Accept sensor id from id/sensorId/sensorID/sensor_id/SN/sn/tagId/deviceId/mac.
- Accept humidity from humi/humidity/Humidity/H, RSSI from rssi/RSSI, battery from bat/battery/Battery/BAT/batt.
- Parse recordedAt/time/timestamp/rtc/RTC, including compact YYMMDDHHmmss.
- When forwarding HTTP gateway readings, use sensor RTC only if within +/-2 hours of server time; otherwise use server receipt time.
- Store unmatched rows with reason invalid_temp, disabled, or unregistered.

TCP GATEWAY LISTENER
- Start a TCP server on LORA_TCP_PORT, default 4001.
- Protocol: TZONE RD07 WiFi TCP frame format v2.0.
- Parse frames starting with TZ bytes, length, payload, CRLF.
- Decode TAG07/TAG09/TAG08/TAG08B records.
- Forward decoded rows through same mapping and forwardToTempMon pipeline.
- Maintain in-memory logs for /templog/api/lora/tcp-log.
- Gateway setup instructions in UI should mention TZConfig.exe -> Data Transfer Protocol -> TCP/IP -> server IP/port.

ALERTING BEHAVIOR
- For chiller/freezer/ambient units, evaluate critical range only:
  - value < criticalMin => critical_low
  - value > criticalMax => critical_high
  - in-range => auto-resolve open critical and legacy warning alerts for that unit
- For warmers, do not create critical_high/low range alerts. Warmers use warmer state machine only.
- Skip creating alerts when unit.inUse === false.
- Prevent duplicate open/acknowledged alert for same unit+type.
- Use unit.alertThresholdMinutes as sustained-excursion delay before creating critical alert in /api/tempmon/ingest path.
- Use reading recordedAt timestamp for the delay, not server arrival time.
- Store in-memory global._tempmonExcursionStart by key `${unitId}_${alertType}`.
- Clear excursion timer when reading returns in range.
- On server startup, re-seed excursion timers by scanning recent readings for active non-warmer units with alertThresholdMinutes > 0 and no open critical alert.
- Device offline check runs every 5 minutes. If active device.lastSeenAt is older than 2 hours and unit is in use, create device_offline alert and send push. Resolve offline alert when a new reading arrives.
- Pending push safety-net runs every 60 seconds for open/acknowledged critical alerts with pushSentAt null.
- On startup, auto-resolve stale warning_high/warning_low alerts because warning alerts are legacy.

WARMER STATE MACHINE
Implement updateWarmerState(unit, device, value, readingTs):
- No-op unless unit.type === 'warmer'.
- Persist state under unit.warmerState.state and unit.warmerState.since.
- Maintain in-memory global._warmerState[unitId] with rolling history.
- Defaults:
  - roomTempCeiling 35 C
  - targetLow = unit.criticalMin
  - offConfirmMinutes 20
  - faultMinutes 30
  - slopeWindowReadings 8
  - riseMinPerMin 0.10 C/min
  - fallMinPerMin 0.08 C/min
- Compute smoothed slope from oldest to newest reading in rolling window.
- Zones:
  - Active zone: value >= targetLow => active.
  - Room zone: value <= roomTempCeiling. If rising => warming_up. If state already off => off. Otherwise cooling until offConfirmMinutes elapsed, then off.
  - Mid zone: roomTempCeiling < value < targetLow. If rising => warming_up unless already fault. If falling => cooling. If flat while warming_up or active for faultMinutes => fault. If flat from cooling/off/unknown => cooling.
- On transition to fault and unit.inUse !== false, create warmer_fault alert and push.
- On leaving fault, auto-resolve open warmer_fault alert with resolveNote.

SECURITY REMINDERS FOR RESTORE
- Never commit actual values for JWT_SECRET, VAPID_PRIVATE_KEY, CLOUDINARY_API_SECRET, MongoDB URI passwords, GATEWAY_API_KEY, or LORA_HTTP_TOKEN.
- If real VAPID private keys or other secrets exist in old docs or git history, rotate them before production use.
- In production, always set JWT_SECRET to a long random value.
- Set GATEWAY_API_KEY or LORA_HTTP_TOKEN for any public ingest endpoint.
- Keep calibration certificates in Cloudinary or equivalent persistent storage; Railway filesystem is ephemeral.
